**⚠️ Never write the actual passcodes into memory** — the memory dir mirrors to git and pushes to GitHub, so any value here = a permanent leaked secret in history (see [[pending-secret-rotation]]). Route passcodes via the clipboard, never the chat/repo.

**Login flow.** `/installer.html` has one **passcode** field → saved in the browser's `localStorage` as `ty_installer_token` → sent as the `x-installer-token` header on every roster/close-out call. Server `lib/installer-auth.js` `resolveInstaller` matches it against the **`INSTALLER_TOKENS`** Netlify env var (JSON map `{"aaron","noah","cody"}`, fail-closed). A valid token scopes the installer to only their own bookings; wrong/missing → 401 → the page clears the stored token and reloads. Set 2026-07-01; as of 2026-07-04 all three are set, each a random **12-char** passcode (owner didn't pick them → doesn't recognize them; that's expected).

**Source of truth = Netlify.** Don't store the values anywhere else — retrieve on demand and drop on the clipboard (owner pastes into the field). Keys only, no leak:
`netlify env:get INSTALLER_TOKENS | node -e '…JSON.parse→Object.keys…'`. To hand the owner one passcode: read INSTALLER_TOKENS in **PowerShell**, `ConvertFrom-Json`, `Set-Clipboard -Value $m.aaron` — print only a length/confirmation, never the value.

**Rotation / reset** (if leaked, or to set a memorable one): `netlify env:set INSTALLER_TOKENS '{"aaron":"…","noah":"…","cody":"…"}'` + redeploy; that instantly invalidates the old token on every device. Capture the owner's chosen value from the clipboard (`Get-Clipboard`), never from chat. For the durable copy the owner wants for "easy reference," the right home is their **password manager**, not memory/git. Related console details: [[event-reminders-automation]].

**⚠ BUSINESS RULE — walk-ins are EVERYDAY, not event-only (owner-stated 2026-07-12).** Clients call in at any moment BETWEEN event dates. So a walk-in/tune can happen (a) AT a scheduled event, AND (b) as everyday business on ANY day the shop takes work. The console MUST let an installer log a walk-in on any date, not only scheduled-event days. **Why:** installers (esp. Cody) do tunes on non-event days; those must be captured so the customer gets a cert and the installer gets OTT commission. **How to apply:** the walk-in create path must NOT require a scheduled event (`installer-walkin.js` previously rejected non-event dates with `unknown-event` — that guard blocks everyday walk-ins). Provide a top-level "log a walk-in any day" entry in `/installer.html` in addition to the per-event adder. Also: the app's notion of "today" must be the installer's LOCAL (Central) date, not server UTC — UTC rolls to tomorrow in the evening, which would mis-date an evening walk-in and push it into the wrong OTT month. See [[monthly-ott-calibration-report]] (Calibration Date = Event Date buckets the commission month). **IMPLEMENTED & LIVE 2026-07-12 (commit 6e3f8dc):** `installer-walkin.js` dropped the `unknown-event` guard (any valid date, defaults to today); `/installer.html` has a top-level "＋ Log a walk-in / call-in" form (name/vehicle/phone + date default today + market picker) beside the per-event adder; `STATE.today = localToday()` (device-local). Everyday walk-in flows through the SAME close-out → cert + OTT report as an at-event one. Verified live: a walk-in on non-event date 2026-07-20 returned `booked` (then deleted).

**Admin role (LIVE 2026-07-12, commit 71c449a).** A new **`INSTALLER_ADMINS`** Netlify env var (comma-separated keys, currently just `aaron`) grants cross-installer access. `lib/installer-auth.js` `isAdmin(key,env)` gates it (fail-closed when unset). An admin: roster returns EVERY installer's bookings (filter drops the `{Installer}=` clause, each booking tagged with its owner `.installer`); walk-in can be added to ANY market (routes to that market's owning installer, not the admin); close-out/no-show works on ANY booking (cert + waitlist still route to the OWNING installer, never misattributed). Console shows an "Admin — viewing" installer filter + per-card installer label when `roster.admin===true`. **New installers are auto-included** (admin sees all) — onboarding an admin = one env edit, no code. Verified live: Aaron's token → `admin:true`, 67 bookings across aaron/cody/noah.

**Smart-feed dashboard redesign (LIVE 2026-07-12, commit bca2455; spec `docs/superpowers/specs/2026-07-12-installer-dashboard-design.md`).** Replaced the month-browser with a time-ordered feed: **Needs close-out → Today → Upcoming → Recent**, each EVENT one `<details>` card showing its whole day together (open jobs → no-shows → a muted "✓ Done" group) with a `done · open` progress pill. Sticky header = running **tally** (this-month done/open/no-show + **lifetime tunes** + **Next:** event) + a **search box that spans ALL history** (name/VIN/vehicle/city/calibration → grouped "Search results"). Per-event **expand state persists** (`STATE.eventOpen`) so a close-out re-render no longer collapses the event you're in. The walk-in adder moved **into each event card** (no dropdown). **Roster now also returns `events`** = the installer's scheduled future events (all markets if admin) so the feed shows upcoming days with zero bookings + accepts the first walk-in (best-effort; a fetch failure just yields `events:[]`). Rendering is client-side over data the roster already sends. Verified live: Aaron's token → 42 future events; browser-verified all sections/search/filter/persistence with mock data.

**Walk-in "goes away" fix (2026-07-12, commit 71c449a).** Cody reported walk-in adds vanishing on `/installer.html`. Root cause was NOT auth (his token was valid) — `addWalkin()` succeeded but gave no confirmation and `renderAll()` collapsed the add-form, so the (actually-saved) entry looked lost. Fixed: persistent green success msg (via `#msg`, which lives outside the re-rendered area), the "+ Add a walk-in" accordion stays open (`STATE.walkinOpen`) for the next add, and error messages are now legible. Lesson: a "form goes away" report ≠ passcode/401; check the success-path re-render first.
